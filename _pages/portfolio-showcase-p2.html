Harmony Design System (Bot Behaviour Design)

  • Low-code platform that creates components/behaviours according to a detailed entity diagram
  • Behaviours had to cater to meet minimal requirements but also allow flexibility in customisation as there were many different type of end-product users
  • The style library (Harmony/SCSS) and DOM structure of components had to be shared across two front-end stacks and also be accessible
  • Constant product iteration from user feedback
  • Balancing as a Product Designer and development

Understand

  • User types: developers that require fast repetitive building blocks created according to their business requirements
  • Who will be using the front-facing end of these building blocks - even though it's for developers to quickly build applications, these features had to be usable by the generic public
  • What would the ideal base requirements for a feature be? But also allow flexibility
  • Throughout the whole project, there were 4 behaviours required: Timelines, CRUD, Forms and Workflows
  • Discovery interviews were done for both developers and their users/clients
  • We were able to identify what were the minimum requirements and what would be nice to have
  • We tried to identify what the most common workflows are for these users for each behaviour

Research & Test

  • We were seeing what common patterns and user experience were being developed by other industries for these behaviours
  • The ideal user journeys for each behaviours were created
  • Different types of workshops were performed between developers and potential behaviour users (developer's client)
  • Minimum requirements were identified from these workshops
  • Rough sketches were done pairing the minimum requirements and the ideal user journey identified - these were constantly re-iterated within the team
  • Sketches were then presented to users for feedback on the flow and the features
  • After re-iterating sketches and refining the main user flow and minimum requirements - sketches turned into wireframes
  • Refine wireframes with user testing with multiple user types

Design and development

  • Harmony had to strictly accommodate to accessibility design and also allow a neutral tone and feel
  • A rough design system was created, mainly for how we want primary and secondary components to look like
  • These components were consistently iterated throughout the hi-fi designs of the behaviour features and screens
  • Once the hi-fi prototype was completed - user testing was done and these designs were refined
  • From here I had to write the DOM structure for each behaviour to ensure both bots share the same standards to allow sharing of a SCSS styling library
  • I then wrote the SCSS library that styled these behaviours and had to write onboarding articles on how to style the bots
  • Given WorkingMouse were using these bots everyday for client-facing work, I became the company's styling the bot mentor for multiple developers
